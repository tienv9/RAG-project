# RAG Study Assistant — project context

## What this is
A RAG (Retrieval-Augmented Generation) app that lets users upload any PDF
and ask questions about it. Answers are grounded in the document with source citations.

## My background
- New CS grad from EWU, strong in full-stack (React, Vue, ASP.NET Core, AWS)
- ML minor — understand embeddings, vectors, cosine similarity, backprop
- Building this as a portfolio project to show AI/ML depth on resume
- First RAG project — learned the concepts from scratch during this build

## How RAG works (my mental model)
1. PDF → extract text + hyperlinks → chunk into ~500 word passages with 50 word overlap
2. Each chunk → embedding model → vector (384 numbers representing meaning)
3. Store vector + original text in ChromaDB under a per-session collection
4. At query time: embed the question → find closest chunk vectors → pack into LLM prompt → answer

Key insight: similar meaning = close vectors. "cat" and "feline" end up near each other
in vector space because the embedding model learned this from billions of sentences.

## Current stack
- **Embeddings**: `all-MiniLM-L6-v2` (HuggingFace, free, 80MB, runs on CPU)
- **LLM**: Ollama — `llama3.2` (local, free, change `OLLAMA_MODEL` in `rag.py` to swap models)
- **Vector DB**: ChromaDB (local, persistent, per-session collections)
- **API**: FastAPI (Python)
- **Frontend**: React (Vite) with react-markdown for rendering LLM output
- **Deploy target**: AWS

## API endpoints
All endpoints require the `X-Session-ID` header (UUID generated by the browser).

- `POST /ingest`      — upload PDF, returns chunk count; 409 if already ingested
- `POST /query`       — ask question, returns answer + sources + chunks_used
- `GET  /documents`   — list documents for this session; also used to verify session exists
- `DELETE /documents` — clear all documents for this session

## Session isolation
Each browser generates a UUID on first visit (`crypto.randomUUID()`) stored in `localStorage`
as `rag_session_id`. Every API request sends it as `X-Session-ID`. The backend namespaces
ChromaDB collections as `documents_<uuid>` so each user's documents are fully isolated.

To inspect or wipe collections manually:
```powershell
cd "Backend FastAPI"
# list all collections
.\venv\Scripts\python -c "import chromadb; c = chromadb.PersistentClient(path='./chroma_db'); print(c.list_collections())"

# delete a specific session collection
.\venv\Scripts\python -c "import chromadb; c = chromadb.PersistentClient(path='./chroma_db'); c.delete_collection('documents_<uuid>')"

# delete a specific file from a session
.\venv\Scripts\python -c "import chromadb; c = chromadb.PersistentClient(path='./chroma_db'); col = c.get_collection('documents_<uuid>'); col.delete(where={'source': 'file.pdf'})"
```

## Shell
Always use the **PowerShell tool** for commands — never the Bash tool. The user is on Windows.
- Use `.\venv\Scripts\python` not `python` or `source venv/bin/activate`
- Use `;` to chain commands, not `&&`
- Use `$env:VAR` for environment variables, not `export VAR=`

## How to run
```powershell
# Ollama must be running and model pulled first
ollama pull llama3.2
# ollama serve  (runs as a background service after install — usually not needed)

# backend
cd "Backend FastAPI"; .\venv\Scripts\pip install -r requirements.txt
.\venv\Scripts\uvicorn main:app --reload --port 8000

# frontend
cd "Frontend React"; npm run dev

# tests
cd "Backend FastAPI"; .\venv\Scripts\python -m pytest test_rag.py -v
```
