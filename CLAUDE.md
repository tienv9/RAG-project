# AskMyDocs — project context

## What this is
A RAG (Retrieval-Augmented Generation) app that lets users upload any PDF
and ask questions about it. Answers are grounded in the document with source citations.

## My background
- New CS grad from EWU, strong in full-stack (React, Vue, ASP.NET Core, AWS)
- ML minor — understand embeddings, vectors, cosine similarity, backprop
- Building this as a portfolio project to show AI/ML depth on resume
- First RAG project — learned the concepts from scratch during this build

## How RAG works (my mental model)
1. PDF → extract text + hyperlinks → chunk into ~500 word passages with 50 word overlap
2. Each chunk → embedding model → vector (384 numbers representing meaning)
3. Store vector + original text in ChromaDB under a per-session collection
4. At query time: embed the question → fetch top-12 candidate chunks by cosine similarity → re-rank with cross-encoder → top 3 go into LLM prompt → stream answer back token by token

Key insight: similar meaning = close vectors. "cat" and "feline" end up near each other
in vector space because the embedding model learned this from billions of sentences.

Re-ranking insight: the bi-encoder (embedding model) is fast but imprecise — it embeds
question and chunk separately. The cross-encoder reads them together and scores relevance
much more accurately. Use bi-encoder to cast a wide net, cross-encoder to pick the best.

## Current stack
- **Embeddings**: `all-MiniLM-L6-v2` (HuggingFace, free, 80MB, runs on CPU)
- **Re-ranker**: `cross-encoder/ms-marco-MiniLM-L-6-v2` (HuggingFace, free, CPU)
- **LLM**: Ollama — `llama3.2` (local, free, change `OLLAMA_MODEL` in `rag.py` to swap models)
- **Vector DB**: ChromaDB (local, persistent, per-session collections)
- **API**: FastAPI (Python), streaming via `StreamingResponse` + SSE
- **Frontend**: React (Vite) with react-markdown for rendering LLM output; streaming fetch with AbortController
- **Deploy target**: AWS EC2 + Docker

## API endpoints
All endpoints require the `X-Session-ID` header (UUID generated by the browser).

- `POST /ingest`        — upload PDF, returns chunk count; 409 if already ingested
- `POST /query/stream`  — ask question; returns SSE stream of `{"type":"token","content":"..."}` events, terminated by `{"type":"done","sources":[...],"chunks_used":[...]}`
- `GET  /documents`     — list documents for this session; also used to verify session exists
- `DELETE /documents`   — clear all documents for this session

## Session isolation
Each browser generates a UUID on first visit (`crypto.randomUUID()`) stored in `localStorage`
as `rag_session_id`. Every API request sends it as `X-Session-ID`. The backend namespaces
ChromaDB collections as `documents_<uuid>` so each user's documents are fully isolated.

To inspect or wipe collections manually:
```powershell
cd "Backend FastAPI"
# list all collections
.\venv\Scripts\python -c "import chromadb; c = chromadb.PersistentClient(path='./chroma_db'); print(c.list_collections())"

# delete a specific session collection
.\venv\Scripts\python -c "import chromadb; c = chromadb.PersistentClient(path='./chroma_db'); c.delete_collection('documents_<uuid>')"

# delete a specific file from a session
.\venv\Scripts\python -c "import chromadb; c = chromadb.PersistentClient(path='./chroma_db'); col = c.get_collection('documents_<uuid>'); col.delete(where={'source': 'file.pdf'})"
```

## Shell
Always use the **PowerShell tool** for commands — never the Bash tool. The user is on Windows.
- Use `.\venv\Scripts\python` not `python` or `source venv/bin/activate`
- Use `;` to chain commands, not `&&`
- Use `$env:VAR` for environment variables, not `export VAR=`

## Deployment plan (EC2 + Docker)

### LLM strategy for prod
Ollama runs locally for free but requires a GPU/CPU-heavy EC2 instance to be usable.
For online deployment, replace Ollama with an API call — much cheaper at low traffic:
- **OpenAI** `gpt-4o-mini` — cheapest capable option (~$0.15/1M input tokens)
- **Anthropic** `claude-haiku-4-5` — similar price tier, strong at instruction-following
- Switch is one function change in `rag.py` (`query_stream`). Abstract behind an env var:
  `LLM_PROVIDER=ollama|openai|anthropic` so local dev still uses Ollama.

Embeddings (`all-MiniLM-L6-v2`) and re-ranker (`ms-marco-MiniLM-L-6-v2`) stay local
inside the container — they're CPU-only, fast, and free. No reason to pay for an embeddings API.

### Docker setup
Two containers via `docker-compose`:
- **backend** — Python/FastAPI image, mounts ChromaDB EBS volume for persistence (no RDS)
- **frontend** — Nginx serving the Vite build, proxies `/api` to backend

```
docker-compose.yml
Dockerfile.backend     # python:3.11-slim, pip install, uvicorn
Dockerfile.frontend    # node build stage → nginx serve stage
```

### EC2 sizing
- `t3.medium` (2 vCPU, 4GB RAM) — enough for embeddings + re-ranker on CPU with API LLM
- `t3.small` if traffic is very low; avoid if RAM pressure causes OOM on model load
- Attach an EBS volume for ChromaDB persistence (survives instance stop/start)

### Rough deployment steps
1. Dockerize backend and frontend locally, verify with `docker-compose up`
2. Launch EC2 (Amazon Linux 2023), install Docker + Docker Compose
3. Push images to ECR (or copy compose file + pull from Docker Hub)
4. Set env vars via AWS SSM Parameter Store: `OPENAI_API_KEY`, `LLM_PROVIDER`, etc.
5. Open ports 80/443 via security group; point domain via Route 53 or just use the EC2 public IP
6. (Optional) Add an Application Load Balancer + ACM cert for HTTPS

### Cost estimate (light portfolio traffic)
- `t3.medium` on-demand: ~$30/month
- EBS 20GB gp3: ~$1.60/month
- OpenAI API at demo traffic: <$1/month
- Total: ~**$33/month**, or ~$20/month with a 1-year reserved instance
